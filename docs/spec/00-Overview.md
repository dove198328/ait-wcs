# 00 - Overview

## 1. 目标
构建一个可平台化交付的 WCS 系统：
- 统一对接多种设备通信（S7、Modbus、以及第三方平台如 AGV RCS 的 HTTP）
- 任务通过工作流推进（Camunda 7.19，嵌入式部署，不单独部署）
- 支持多仓库：以 `warehouseId` 作为仓库唯一标识
- 支持多仓型：立体库（堆垛机 ASRS）、密集库（四向车 Shuttle）、混合仓
- 支持能力可插拔：同入同出、换巷道、双深位临时移库、异常处理策略等

## 2. 非目标（明确不做）
- 不负责 WMS/OMS 的业务建模与订单拆分（只接收 WMS 下发的 WCS 任务或更细粒度指令计划）
- 不把设备协议细节放入 BPMN（BPMN 只做编排与决策，不直接做点位读写）
- 不实现通用 3D 数字孪生（可提供事件/状态输出给大屏，但不内置复杂仿真引擎）

## 3. 关键约束
- Camunda 引擎嵌入在 WCS 的 Spring Boot 进程中（同 JVM）
- 多仓库强约束：任何流程部署/启动/任务拉取/查询必须绑定 `warehouseId`，否则视为严重缺陷
- 设备执行必须幂等：重复拉取/重试不得导致重复写入产生不可控动作
- 所有可变点必须“配置化”或“插件化”，禁止在核心流程里出现大量项目 if-else

## 4. 概念分层（模块 / 插件 / 责任链节点）
- L1 模块（Module）：独立域能力，可按项目选装（ASRS 域、Shuttle 域、RCS 集成域等）
- L2 插件（Plugin）：模块内的可选能力（同入同出、换巷道、双深位临时移库、异常处理策略）
- L3 责任链节点（Policy/Step）：插件内部或框架内部的最细粒度步骤，可配置顺序与启用

## 5. 运行时主数据流（文字时序）
1) 上游（WMS/人工）通过 API 提交任务（包含 warehouseId）
2) 进入 StartCoordinator：按 warehouseId 建立 WarehouseContext，并进行 StartGate 门禁判断
3) 允许启动：以 businessKey（warehouseId+upstreamId+taskType）幂等启动流程实例
4) 流程推进到 External Task（或嵌入式 Worker 处理节点）：
   - 生成执行计划 Plan（可插入临时移库等扩展）
   - 执行计划中的 CommandPipeline（锁→下发→等待反馈→落库→回写流程）
5) 完成/失败/业务异常：回写流程，并更新任务状态、指令审计、告警事件

## 6. 验收标准（高层）
- 多仓库：两个 warehouseId 的流程、任务、审计记录绝不互相可见（除超级管理员）
- 基础任务：出/入/盘/移四类任务可跑通一条 E2E
- 插件：同一套代码在不同 warehouse 的 profile 下启用/禁用同入同出、换巷道、双深位临时移库，行为符合配置
- 异常：放货有货、取货无货可触发并按策略进入挂起/纠偏/换位/人工处理流程
